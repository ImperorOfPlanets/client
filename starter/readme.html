Инструкция по работе со starter.py

Основные режимы работы

Скрипт starter.py поддерживает два основных режима работы:

1. Интерактивный режим (по умолчанию)

Запускается при обычном выполнении скрипта без аргументов:

python starter.py

Что делает:

  • Проверяет IP-адреса системы
  • Проверяет наличие обновлений
  • Проверяет установку Docker и Docker Compose
  • Проверяет структуру папок
  • Отображает интерактивное меню с возможностями настройки

Меню интерактивного режима:

=== ГЛАВНОЕ МЕНЮ ===
1. Собрать образы и отправить в реестр
2. Обновить образы из реестра
3. Управление службой
4. Проверить обновления
5. Показать справку
0. Выход

2. Режим службы

Запускается с аргументом --service-run:

python starter.py --service-run

Что делает:

  • Работает в фоновом режиме
  • Выполняет задачи по расписанию (из SCHEDULER_CONFIG)
  • Проверяет состояние Docker контейнеров
  • Логирует все действия в файлы журналов

Логика работы скрипта

Проверки при запуске:

  • Лимит запусков - не более MAX_RUNS_PER_PERIOD (5) запусков за PERIOD_MINUTES (5 минут)
  • Режим работы - определяется по аргументам командной строки
  • Системная информация - собирается и логируется

Основные функции:

  • Проверка Docker (check_docker_installed, check_docker_compose_installed)
  • Управление контейнерами (run_docker_compose)
  • Проверка обновлений (start_updates_projects)
  • Управление службой (manage_service)
  • Логирование (класс ProjectLogger)

Особенности работы:

  • При первом запуске создает необходимые папки (docker, laravel)
  • Для работы с Docker автоматически генерирует docker-compose.yml из шаблона
  • Поддерживает отправку образов в реестр и их обновление
  • Имеет встроенную систему защиты от частых перезапусков

Улучшения:

  • Можно добавить валидацию конфигурации при старте
  • Реализовать более надежное управление службой (через systemd/supervisor)

Безопасность:

  • Все пути нормализуются через Path
  • Есть проверка лимитов запусков
  • Логирование содержит достаточную информацию для диагностики

Стабильность:

  • Хорошая обработка ошибок в критических местах
  • Есть резервное копирование перед обновлениями
  • Логирование помогает отслеживать проблемы

Рекомендации: Скрипт готов к использованию в обоих режимах. Для production-использования рекомендуется настроить его как службу через --service-run и управлять через системные инструменты (systemd/supervisor).
